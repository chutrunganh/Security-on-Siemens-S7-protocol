\paragraph{Historical attack chain.}

\begin{enumerate}
    \item \textbf{Windows infection:} Stuxnet entered internal networks via USB and spread using Windows zero-days, reporting to C\&C servers~\cite{falliere2010stuxnet,stuxnet_work_details}.
    \item \textbf{Step~7 hooking (*):} On each infected workstation it hooked \texttt{s7otbxdx.dll} to intercept and modify traffic between Step~7 and the PLC---a software-layer man-in-the-middle attack (Figures~\ref{fig:stuxnet_mitm_sim-image-9} and~\ref{fig:stuxnet_mitm_sim-image-10}).
    \item \textbf{Centrifuge control interference:} Stuxnet modified PLC logic to raise centrifuge speed above safe limits. Malicious code was injected via Step~7, executed on the PLC, forwarded to the variable frequency drive (VFD), and physically damaged the centrifuge while falsified values were sent to WinCC (Figure~\ref{fig:stuxnetmitmsimimage11png}).
\end{enumerate}

Stuxnet activated only when the target PLC and drive matched narrow criteria: Siemens S7-300, Vacon or Fararo Paya drives, and operating frequency between 807~Hz and 1210~Hz---typical of gas centrifuges rather than ordinary industrial equipment. When matched, it stepped output frequency through levels such as 1410~Hz, 2~Hz, and 1064~Hz to damage the process while concealing the effect on monitoring data.

\textit{In this thesis, the same operational outcome is reproduced at the S7comm network layer (Figure~\ref{fig:lab-topology}) rather than via DLL hooking as in the historical campaign.}

(*) \textit{TIA Portal bundles STEP~7, WinCC, PLCSIM, and drive configuration tools for Siemens S7 PLC families.}

\paragraph{Attack simulation.}

\textit{The goal is to change a PLC datablock variable (altering process behaviour) while intercepting HMI-bound responses so the operator display remains unchanged.}

\subsubsection{Attacker}

Three steps run in parallel on the attacker VM:

\begin{enumerate}
    \item \textbf{Unauthorized write:} Using Snap7 or Python S7 Comm (\url{./attacker/test_readwrite_db_via_snap7.py} or \url{./attacker/test_readwrite_db_via_pythonS7comm.py}), write \texttt{DB1.DBW2} (\texttt{iSetpoint}) from 1200 to 2000. S7comm has no authentication~\cite{inprotech_s7comm_security}.
\end{enumerate}

\begin{packetdump}
    Current Speed  | Setpoint Speed    | Centrifuge Status
    --------------------------------------------------
    b'\x03\xc1'    |    b'\x04\xb0'    |    b'\x00'
    b'\x07\xe4'    |    b'\x07\xd0'    |    b'\x01'
\end{packetdump}

\texttt{iSetpoint} changes from \texttt{0x04b0} to \texttt{0x07d0}; \texttt{bCentrifugeStatus} becomes 1. Without MITM, the HMI shows oscillation around 2000 in red (Figure~\ref{fig:stuxnet_mitm_sim-image-14}) and PLC-to-HMI packets carry the new setpoint (Figure~\ref{fig:stuxnet_mitm_sim-image-15}).

\subsubsection{Attack analysis}

Packet editing at line rate requires redirecting traffic (Bettercap ARP spoof) and queuing TCP/102 flows in NFQUEUE because Bettercap's JavaScript editing cannot host a custom S7 parser. The intercept script enables IP forwarding, relaxes reverse-path filtering, adds a route to the PLC subnet, and traps port~102 in NFQUEUE via \texttt{iptables} (\url{./attacker/mitm/s7_intercept.py}).

The S7 PDU layers are TPKT, COTP, S7 header, then Param/Data (Figure~\ref{fig:iemensS7protocolstackpng}). Figures~\ref{fig:stuxnet_mitm_sim-image-19} and~\ref{fig:stuxnet_mitm_sim-image-20} show TPKT and COTP headers. The S7 header starts with \texttt{0x32}; byte~8 (\texttt{ROSCTR}) distinguishes Job (\texttt{0x01}), Ack (\texttt{0x02}), and Ack\_Data (\texttt{0x03})---see Read Var request/response examples in Figures~\ref{fig:stuxnetmitmsimimage25png} and~\ref{fig:stuxnetmitmsimimage26png}.
