\section{Research methodology overview}
\label{sec:methodology-overview}

This chapter presents the experimental methodology of the thesis: building a simulated industrial control system environment, reproducing typical attack behaviors on the S7comm protocol, analyzing the captured network traffic, and creating a basis for building detection rules with Suricata in later chapters.

The research process is organized into five consecutive steps as follows.

\begin{enumerate}
    \item Study and deployment of simulation: deploy the simulation environment on VMware virtualization software using virtual machines:
    \begin{itemize}
        \item Simulated PLC via OpenPLC Runtime v4
        \item HMI via FUXA
        \item Ubuntu Desktop/Server for the attacker machine, router, and IDS.
    \end{itemize}

    \textit{In some scenarios, TIA Portal + S7-PLCSIM Advanced + WinCC is also used to simulate the PLC and HMI because of simulation limits in OpenPLC Runtime v4.}
  
    \item Building attack modules: implement Python scripts and a GUI to run reconnaissance, Start/Stop command injection, upload/download operations, DoS and malformed packets on the S7comm channel, and man-in-the-middle attacks on the S7comm protocol.
    
    \item Traffic collection and analysis: capture network traffic with Wireshark, analyze S7comm packet structure, and create a basis for building detection rules with the Suricata IDS.
    
    \item Detection rule development: convert observed indicators into Suricata signatures based on byte offsets, function codes, frequency thresholds (DoS), and abnormal syntax (malformed).
    
    \item IDS testing: run Suricata on traffic copies and compare alerts with each simulated scenario.
    
\end{enumerate}

\section{Study and deployment of simulation}
\label{sec:lab-setup}

\subsection{Deployment principles}

The lab environment is designed as a controlled simulation on virtual machines. All virtual machines are deployed using VMware LAN Segment objects on three separate segments:
\begin{itemize}
    \item \textbf{Control segment} \texttt{192.168.50.0/24} (\texttt{CONTROL\_NET}): the controller layer, with OpenPLC at \texttt{192.168.50.10} listening on TCP/102;
    \item \textbf{Supervisory segment} \texttt{192.168.60.0/24} (\texttt{SUPERVISOR\_NET}): the authorized HMI/engineering layer, with FUXA at \texttt{192.168.60.10}, the only legitimate origin of S7comm management traffic;
    \item \textbf{Attacker segment} \texttt{192.168.70.0/24} (\texttt{EXTERNAL\_NET}): an isolated LAN hosting the Attacker VM at \texttt{192.168.70.20}, placed \emph{outside} the supervisory zone so that attack traffic is distinguishable from normal engineering traffic by source address.
\end{itemize}
A virtual router at \texttt{192.168.50.1}/\texttt{192.168.60.1}/\texttt{192.168.70.1} routes between the segments (Figure~\ref{fig:lab-topology}). The IDS VM shares the Controller segment in promiscuous mode. Placing the attacker in its own segment is what allows the detection rules of Chapter~\ref{sec:detection-overview} to use the source \texttt{!\$SUPERVISOR\_NET} (Section~\ref{sec:rule-source-zones}): commands reaching the PLC from outside the supervisory zone are treated as suspicious, while the same commands from \texttt{SUPERVISOR\_NET} are considered normal.


\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.95\textwidth]{Figure/lab_topology.png}
    \caption{Lab topology on VMware LAN segments}
    \label{fig:lab-topology}
\end{figure}


\textit{Because VMware vSwitch behaves like a conventional hub rather than a switch, placing the IDS in the same LAN segment and enabling promiscuous mode on the network interface is equivalent to a SPAN/mirror port on a physically deployed switch.}

\subsection{IDS deployment}
\label{sec:ids-deployment}

Network IDS can be placed \emph{inline} on the traffic path (often operating as an IPS when blocking is enabled) or \emph{passively} on a mirrored copy of the same segment (SPAN port, TAP, or promiscuous capture as in this lab). Table~\ref{tab:ids-inline-vs-passive} compares the two modes on criteria relevant to OT networks. Passive deployment is preferred in this thesis because industrial control prioritises availability and deterministic response times on the PLC communication path~\cite{nist_sp800_82}.

\begin{table}[H]
    \centering
    \caption{Inline versus passive IDS deployment (selected criteria)}
    \label{tab:ids-inline-vs-passive}
    \begin{tabular}{|p{3.1cm}|p{4.9cm}|p{4.9cm}|}
        \hline
        \textbf{Criterion} & \textbf{Inline (IPS)} & \textbf{Passive (SPAN/TAP)} \\
        \hline
        Impact on control traffic &
        Sensor is in the data path; every packet passes through the engine, adding latency and a potential bottleneck &
        \textbf{No change to live traffic}; only a copy is analysed, suitable when production must not be disturbed \\
        \hline
        Risk if the sensor fails &
        Misconfiguration, overload, or hardware fault can interrupt or delay PLC/HMI sessions on that link &
        \textbf{Sensor failure does not break the control path}; worst case is loss of visibility, not loss of connectivity \\
        \hline
        Deployment on existing plant &
        Often requires re-cabling, new failure domains, and change windows on critical links &
        \textbf{Can be added beside the segment} via mirror/TAP without moving PLC or HMI endpoints \\
        \hline
        Real-time blocking &
        Can drop or reset suspicious flows on the wire &
        Detection and alerting only; response is left to operators or external systems \\
        \hline
        Rule tuning and maintenance &
        Aggressive or experimental rules risk blocking legitimate engineering traffic immediately &
        \textbf{Rules can be tested and refined} while observing alerts before any inline enforcement is considered \\
        \hline
    \end{tabular}
\end{table}

In the lab, Suricata runs on a dedicated IDS VM that receives mirrored traffic on \texttt{ens33}; it does not forward, modify, or drop packets on the OT path. Chapter~5 evaluates alert generation under this passive placement.

\subsection{Tools used}

\subsubsection{OpenPLC}

OpenPLC is an open-source platform that simulates a physical PLC on a computer. The software runs on Windows, Linux, macOS, and Raspberry Pi. It supports control programs under the IEC 61131-3 standard~\cite{iec61131_3} and can connect to SCADA, HMI, or other industrial devices. OpenPLC has two main components:

- \href{https://autonomylogic.com/runtime}{OpenPLC Runtime}: the PLC engine that simulates a physical PLC. OpenPLC Runtime V4 supports several industrial protocols, including Siemens S7comm. When this protocol is enabled on OpenPLC Runtime V4, the runtime listens on port 102 and serves the Siemens S7comm protocol through the internal Snap7 driver~\cite{openplc_s7_driver}. This thesis uses OpenPLC Runtime V4.

\subsubsection{Snap7}

Snap7 is an open-source library developed to connect to, transmit data with, and control automation devices. It is one of the most widely used open-source libraries in the automation industry.

Snap7 supports Siemens S7 communication protocols (S7comm) to connect and exchange data with Siemens automation devices. Its main features include:

\begin{itemize}
    \item Reading and writing variables in the PLC
    \item Accessing DB and OB blocks in the PLC
    \item Providing device status information
\end{itemize}

Characteristics of Snap7: 

\begin{itemize}
    \item Can be used with common programming languages such as C++, C\#, Python, and Java, allowing developers to choose the language best suited to their project. 
    \item Runs on multiple operating systems, including Windows, Linux, and macOS, which gives flexibility in application development.
    \item Compatible with many S7 PLC families and supports communication with S7-200, S7-300, S7-400, S7-1200, and S7-1500 PLCs, making integration into existing systems easier.
\end{itemize}

\subsection{Reconnaissance}
\label{sec:reconnaissance}

In the MITRE ATT\&CK for ICS matrix, this scenario maps to the \emph{Discovery} tactic: \textbf{T0846 Remote System Discovery} (locating PLCs on the network) and \textbf{T0888 Remote System Information Discovery} (collecting device model, firmware, and program-block details)~\cite{mitre_ics_attack}. The same reconnaissance steps (network scanning followed by in-depth block enumeration) were demonstrated against real S7-300 controllers by Alsabbagh and Langendoerfer~\cite{alsabbagh2022remote}.

This scenario combines a pre-built scanner with self-written enumeration tools. First, the pre-built tool Nmap is used with NSE scripts to discover devices in a network range. These scripts use S7comm-specific scanning techniques to avoid overloading devices in the OT network.

\subsubsection{Scanning with other tools based on DCP}


DCP is a protocol in the PROFINET standard set~\cite{profinet_dcp}. It is used to discover and configure devices on a PROFINET network. It is mainly used to connect to PLCs running on PROFINET before they can be assigned an IP address, because DCP operates at Layer 2 (Data Link), similar to ARP in TCP/IP. A DCP identify request is sent to discover devices on the network and wait for responses from those devices.


DCP can perform three main functions:

\begin{enumerate}
    \item Discovery: scan all devices connected on the same LAN.
\end{enumerate}


\begin{enumerate}
    \item Basic Configuration: assign IP address, subnet mask, gateway, and NameOfStation to a device.
\end{enumerate}


\begin{enumerate}
    \item Identification: request a specific device to blink its LED so its physical location in the electrical cabinet can be confirmed.
\end{enumerate}


There are several ways to scan PLC devices through DCP:


\begin{itemize}
    \item Pre-built: open TIA Portal and run PLC discovery
    \item Pre-built: use the Metasploit module auxiliary/scanner/scada/profinet\_siemens
    \item Self-written tool: \texttt{SiemensScan.py}, which sends Layer-2 PROFINET-DCP identify requests and collects the responding devices. This mirrors the PNIO~Scanner function of the IHP-Attack tool~\cite{alsabbagh2022remote}, which also discovers S7-300 devices via PROFINET-DCP identify-response packets
\end{itemize}

\subsection{Start/Stop PLC (Unauthorized Command Injection)}
\label{sec:start_stop_plc}


This attack sends an unauthorized \texttt{Start CPU} / \texttt{Stop CPU} command to the PLC over the S7comm channel, which performs no authentication on these control messages. Strictly speaking this is not a replay attack: the Start and Stop PDUs are \emph{fixed, well-known payloads} (the same bytes the TIA Portal always sends, with PI-Service \texttt{P\_PROGRAM}), so the attacker does not need to capture a prior session; it simply re-sends the constant command. Function codes \texttt{0x28} (PLC Control) and \texttt{0x29} (PLC Stop) with PI-Service \texttt{P\_PROGRAM} follow the layout documented for Job/Ack Data messages~\cite{gmiru_s7comm_part2}. This scenario uses the self-written tool \texttt{start\_stop\_plc.py} together with \texttt{payloads.py}, which hold the fixed raw PDUs and send them to the PLC over a plain TCP socket. The identical Start/Stop payloads were reported against real S7-300 controllers by Alsabbagh and Langendoerfer~\cite{alsabbagh2022remote}.

In the MITRE ATT\&CK for ICS matrix, this maps to \textbf{T0855 Unauthorized Command Message} (sending a control command the device should not accept from this source) under the \emph{Impair Process Control} tactic, and to \textbf{T0816 Device Restart/Shutdown} under the \emph{Inhibit Response Function} tactic when the Stop command halts the CPU~\cite{mitre_ics_attack}.

\subsection{DoS attack on the S7comm channel}
\label{sec:dos_s7comm}

In OT environments, DoS aims to disrupt safe operation: the PLC cannot process control commands in time, the HMI loses data synchronization, or the engineering station cannot establish an S7 session. MITRE ATT\&CK for ICS classifies these behaviors under the Impact tactic, for example T0814 Denial of Service, T0803 Block Command Message, and T0804 Block Reporting Message~\cite{mitre_ics_attack}.

DoS against Siemens PLCs over Ethernet typically exploits TCP port 102 and the S7comm protocol:

\begin{enumerate}
    \item \textbf{Connection resource exhaustion:} open/close TCP connections to port 102 at high frequency (connection churn), or send many SYN packets, filling the session table on the PLC/S7 stack.

    \item \textbf{Application-layer overload with valid S7 PDUs:} the attacker sends syntactically correct requests on established sessions; the PLC must parse and queue each one even though no malformed bytes are present. Two variants differ by intent and opcode mix:
    \begin{itemize}
        \item \emph{Generic command flood} (DoS as primary goal): high-rate repetition of \texttt{Setup Communication} (\texttt{0xF0}), \texttt{Read/Write Var}, or mixed Job traffic to saturate the S7 processing queue regardless of what is being read or written.
        \item \emph{Reconnaissance at abusive rate} (inventory gathering with side effect of service degradation): high-rate \texttt{Read SZL} or \texttt{List Blocks}, the same opcodes used in Section~\ref{sec:reconnaissance} for legitimate enumeration but sent fast enough to load the CPU comparably to aggressive OT asset scanning or misconfigured SCADA polling.
    \end{itemize}
\end{enumerate}

This scenario uses the self-written tool \texttt{s7comm\_dos.py}, which generates the connection-churn/SYN flood and the high-rate S7 PDU floods using raw sockets and the Snap7 library. Detection rules for connection exhaustion and for both application-layer variants are defined in Section~\ref{sec:rules-dos}.

\subsection{PLC Program Upload and Download}
\label{sec:down_up_program}

In Siemens terminology, the meaning is opposite to common IT usage:


\begin{itemize}
    \item Download: the master/client sends the control program to the slave PLC. This corresponds to loading code in TIA Portal.
\end{itemize}


\begin{itemize}
    \item Upload: the master/client retrieves the control program from the PLC.
\end{itemize}


The control program source and most program data are stored as blocks. Block types in Siemens devices include OB, (S)DB, (S)FC, and (S)FB as mentioned earlier. In the MITRE ATT\&CK for ICS matrix, uploading the PLC program maps to \textbf{T0845 Program Upload} (\emph{Collection} tactic), and pushing a program back to the PLC maps to \textbf{T0843 Program Download} (\emph{Lateral Movement} tactic); when the downloaded program is altered, it additionally maps to \textbf{T0889 Modify Program} under \emph{Impair Process Control}~\cite{mitre_ics_attack}. \subsection{Sending malformed packets to the PLC}
\label{sec:malformed_plc}

Malformed packets (syntactically or semantically invalid packets) on the S7comm channel target the parser and protocol stack on the PLC or CP, rather than sending valid control commands at high frequency as in DoS (Section~\ref{sec:dos_s7comm}). In practice and in ICS fuzzing documentation, attackers often send PDUs with length fields that do not match the payload, invalid S7 headers, or Job/Userdata parameters with missing bytes, in order to cause temporary denial of service, close TCP/102 sessions, or (in rare cases) exploit memory bugs on weaker stacks.

This thesis does not aim to cover every infinite fuzz variant. Instead, it systematizes representative malformed groups commonly encountered when pentesting or fuzzing S7comm, then selects one or two representative samples per group to build lab payloads and corresponding Suricata rules.

In the MITRE ATT\&CK for ICS matrix, sending malformed PDUs that can crash or hang the protocol stack maps to \textbf{T0814 Denial of Service} (\emph{Inhibit Response Function} tactic), and to \textbf{T0816 Device Restart/Shutdown} when a fault forces the CPU into an error/restart state~\cite{mitre_ics_attack}.

\paragraph{Four commonly used malformed groups.}

\begin{enumerate}
    \item Group 1, Encapsulation (TPKT / COTP): errors at the ISO-on-TCP layer before S7 processing. Examples: TPKT version other than \texttt{0x03}; TPKT \emph{length} field (bytes 2--3) declared larger or smaller than the actual TCP payload; COTP not a standard Data TPDU \texttt{02 F0 80}. The stack usually stops early and cannot parse the Job.
    \item Group 2, S7 header: TPKT/COTP may look valid, but the Protocol ID byte $\neq$ \texttt{0x32} or \texttt{ROSCTR} is not in the valid set (\texttt{0x01} Job, \texttt{0x02} Ack, \texttt{0x03} Ack\_Data, \texttt{0x07} Userdata). This is the layer for which Chapter~\ref{sec:detection-overview} uses offsets 7--8 for opcode rules.
    \item Group 3, Function layer (Job / parameters): a valid S7 header (\texttt{32 01}) but a nonexistent function code (e.g.\ \texttt{0xFF}) or a \texttt{Write Var} PDU (\texttt{0x05}) that is too short and lacks the item structure, so the PLC returns a parameter error (similar to codes \texttt{0x0112}, \texttt{0xD801} in S7 documentation).
    \item Group 4, Truncation: sending only the beginning of a PDU (e.g.\ Setup Communication) without enough bytes to reach the function offset (\texttt{17}). This is a common ``missing-byte'' malformed pattern in random fuzzing or when a proxy truncates packets.
\end{enumerate}

\paragraph{Representative packet modifications.}

Each lab sample starts from a valid Setup Communication PDU (same hex template as in the DoS script) and changes only one field or truncates the PDU so faults are easy to compare in Wireshark. These samples are produced by the self-written tool \texttt{s7comm\_malformed.py}, which builds each structurally invalid TPKT/COTP/S7 PDU and sends it to the PLC.
